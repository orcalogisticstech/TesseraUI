The MVP is a set of three simple decision APIs that help a warehouse decide what work to start, how to group it, and what to do first. The first API, Optimize Release, looks at the current backlog of orders plus basic operating conditions and recommends which work should become active now versus wait until later. The second, Optimize Batching, takes the active work and groups it into efficient work packages, such as batches or clusters, so the floor is not flooded with scattered work. The third, Prioritize Work, ranks the active work packages so the team knows what deserves attention first. This fits how major warehouse systems publicly describe execution: they commonly support work release, task or work-package management, priorities, and assignment-related actions, while deeper control like exact walking routes is much less clearly exposed publicly.

A typical workflow would look like this: every 15 minutes, the warehouse sends Tessera a snapshot of its current state. That snapshot might include open orders, promised ship times, where products are stored, how many workers are available, and what work is already active. Tessera first returns a recommendation such as, “activate these 120 orders now and hold these 40 for the next cycle.” Then the warehouse sends the released work into the batching API, which might return something like, “group these 120 orders into 10 work packages so they can be handled more efficiently.” Finally, the prioritize API ranks those 10 work packages based on urgency and current floor conditions. For example, if one batch is close to a shipping deadline but another would create a traffic jam in a crowded area, Tessera can recommend doing a different batch first. That reflects how real warehouse systems typically dispatch work: priority matters, but it is usually combined with other factors like worker fit, queue state, and operational efficiency rather than followed blindly in strict numeric order.

The inputs and outputs are meant to be easy to understand, even without warehouse expertise. Inputs are things like: a list of orders, due times, item locations, current active work, and simple information about available staff or congestion. Outputs are recommendation lists, not black-box commands: “release now” or “defer,” “put these orders in the same batch,” and “this batch should be worked before that one.” Each API also returns a short explanation and an estimated effect, such as lower travel, less crowding, or lower risk of missing a deadline. So the product is not “replace your warehouse software.” It is more like a decision layer on top of existing systems: a service that continuously helps operations teams activate the right work, package it sensibly, and focus on the most important next step.
